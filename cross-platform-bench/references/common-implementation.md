# Common Implementation

## CMake For Single Bench

In `single-bench-cli` projects, update the generated top-level targets directly. Do not add multi-bench manifest or registry directories unless the generated project already provides them.

```cmake
add_library(<project_slug>_support STATIC
  src/<bench_logic>.cpp
)

target_include_directories(<project_slug>_support
  PUBLIC
    ${CMAKE_CURRENT_SOURCE_DIR}/include
)

target_link_libraries(<project_slug>_support
  PUBLIC
    libxcpu
    RapidJSON
)

add_library(<project_slug>_bench SHARED
  src/main.cpp
  src/<bench_logic>.cpp
)

target_link_libraries(<project_slug>_bench
  PUBLIC
    sbench
    libxcpu
    RapidJSON
)

add_executable(<project_slug>
  src/main.cpp
  src/<bench_logic>.cpp
)

target_link_libraries(<project_slug>
  PRIVATE
    sbench
    libxcpu
    RapidJSON
)
```

Keep tests linked to `<project_slug>_support` when they exercise reusable logic instead of the CLI entrypoint.

## CMake For Multi Bench Leaf

Use project helper functions generated by the benchmark project template when they exist:

```cmake
add_library(instr_tp_support STATIC
  src/instr_tp.cpp
)

target_include_directories(instr_tp_support
  PUBLIC
    ${CMAKE_CURRENT_SOURCE_DIR}/include
)

project_add_bench(instr_tp
  SOURCES
    src/main.cpp
    src/instr_tp.cpp
)

project_bench_include_directories(instr_tp
  PRIVATE
    ${CMAKE_CURRENT_SOURCE_DIR}/include
)

project_bench_link_libraries(instr_tp
  PRIVATE
    libxcpu
    RapidJSON
)

project_install_bench(instr_tp)
```

`project_add_bench` should link `sbench` and define the build mode needed by `sbench/register.hpp`, such as `SBENCH_BUILD_SPLIT_EXE` for a split executable or `SBENCH_BUILD_SINGLETON` for a suite-registered library.

## MicroBench Pattern

Use `MicroBench` for a leaf command:

```cpp
class HelloBench : public MicroBench {
public:
  HelloBench() : MicroBench("hello") {
    app->description("Hello world benchmark smoke command");
    app->add_option("--name", name_, "Name to print");
    app->callback([this]() { run(); });
  }

private:
  std::string name_ = "world";
  void run();
};
```

For framework projects, include `sbench/register.hpp` alongside `sbench/sbench.hpp`.

`SBENCH_BENCH_FACTORY`, `SBENCH_BENCH_REGISTER_INTO`, and `SBENCH_SUITE_ENTRY` are provided by `sbench/register.hpp`. Do not reimplement these macros in a benchmark source file.

## Suite Pattern

Use a bench-local factory for leaf commands, then expose a `ForkableBench` suite through `SBENCH_SUITE_ENTRY`:

```cpp
SBENCH_BENCH_FACTORY(hello_bench_factory);

class HelloBench : public MicroBench {
public:
  HelloBench() : MicroBench("hello") {
    app->description("Hello world smoke bench");
    app->callback([this]() { run(); });
  }

private:
  void run();
};

SBENCH_BENCH_REGISTER_INTO(HelloBench, hello_bench_factory);

class HelloSuite : public ForkableBench {
public:
  HelloSuite() : ForkableBench(hello_bench_factory_get_instance()) {
    app->name("hello_bench");
    app->description("Hello benchmark suite");
  }

protected:
  std::unique_ptr<ForkableBench> create_fork_instance() const override {
    return std::make_unique<HelloSuite>();
  }
};

SBENCH_SUITE_ENTRY(HelloSuite);
```

`SBENCH_SUITE_ENTRY` is intentionally controlled by the build. With `SBENCH_BUILD_SINGLETON`, it registers the suite into the default sbench suite factory; with `SBENCH_BUILD_SPLIT_EXE`, it emits `main()`.

## Hello World Example

A minimal smoke command can print plain text:

```cpp
void HelloBench::run() {
  println("hello {}", name_);
}
```

For benchmark data, prefer JSONL output instead of prose.
